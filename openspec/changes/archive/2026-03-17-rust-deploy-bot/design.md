## Context

当前项目需要实现自动化部署能力，替代人工部署操作。项目基于 Rust 语言开发，需要提供 Webhook 接口接收外部触发信号，实现代码拉取、依赖安装、项目构建的自动化流程。

## Goals / Non-Goals

**Goals:**
- 实现基于 Rust 的 HTTP 服务，提供 Webhook 接口
- 支持 GitHub/GitLab 等主流平台的 Webhook 格式
- 自动识别项目类型并执行对应的构建流程

**Non-Goals:**
- 不实现复杂的用户认证和权限管理（初期仅支持简单 Token 验证）
- 不实现集群化部署（单节点部署模式）
- 不实现部署前的审批流程

## Decisions

1. **Web 框架选择: Axum**
   - 理由：Axum 是 Tokio 官方推荐的 Web 框架，异步高性能，与 Rust 生态契合
   - 替代方案：Actix-web（更重）、Rocket（同步）

2. **任务队列: 内存队列 + 异步处理**
   - 理由：初期部署规模较小，无需引入 Redis 等外部队列
   - 替代方案：Redis/RabbitMQ（未来规模扩大后可迁移）

3. **Git 操作: git2 库**
   - 理由：纯 Rust 实现，无外部依赖，性能好
   - 替代方案：调用系统 git 命令（需要处理进程管理）

4. **配置管理: Config + Serde**
   - 理由：支持多种配置格式（YAML/TOML/JSON），与 Rust生态一致

5. **日志: tracing**
   - 理由：结构化日志，与 async/await 配合良好
   - 替代方案：log + env_logger

## Risks / Trade-offs

- **风险**: 并发部署多个项目可能导致资源竞争
  - **缓解**: 初期限制单项目并发，构建任务队列串行处理

- **风险**: Webhook 安全性
  - **缓解**: 实现 Token 验证，支持 IP 白名单

- **风险**: 构建过程长时间占用
  - **缓解**: 超时控制，支持任务取消

- **权衡**: 部署日志存储在文件中而非数据库
  - 理由：简化实现，便于排查问题
